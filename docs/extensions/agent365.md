# Agent 365 Channel

The Agent 365 channel extension bridges the **Microsoft 365 Agents SDK** `Activity` protocol to the
BotNexus routing pipeline, so an agent's messages can flow in and out of Microsoft 365 surfaces while
**BotNexus remains the response engine**. The SDK is used purely as a channel abstraction — all
responses are generated by the normal BotNexus agent loop and providers.

## Overview

| Property | Value |
|----------|-------|
| Extension ID | `botnexus-agent365` |
| Channel key | `agent365` |
| Source | `BotNexus.Extensions.Channels.Agent365` |
| DI extension | `AddBotNexusAgent365Channel(...)` |
| Config section | `channels:agent365` |
| Tier | Register (message round-trip only) |

## Scope

This adapter implements the **Register tier** (PBI1 of the Agent 365 epic): the agent is discoverable
on a Microsoft 365 surface and a message round-trip works. The following are intentionally **out of
scope** and land in later PBIs:

- Identity blueprint / inbound JWT validation — **PBI3**
- Work IQ — **PBI4**
- Teammate experience — **PBI5**

## How it works

- **Inbound.** Microsoft 365 surfaces POST an `Activity` (a `message`-type activity) to the adapter's
  message endpoint (default `POST /agent365/messages`, hosted by `Agent365InboundController`). The
  activity is translated by `Agent365ActivityTranslator.ToInboundMessage` into an `InboundMessage`
  (sender, conversation, text, image attachments) and dispatched through `IChannelDispatcher`.
- **Outbound.** When the BotNexus loop produces a reply, `Agent365ChannelAdapter.SendAsync` translates
  the `OutboundMessage` into a reply `Activity` (`Agent365ActivityTranslator.ToReplyActivity`) and
  sends it through the Agents SDK connector (`IConversations.ReplyToActivityAsync`), authenticating
  with an Entra client-credential MSAL flow.
- **Channel address.** The conversation id and the reply `serviceUrl` are folded into the opaque
  `ChannelAddress` (`<conversationId>|svc:<serviceUrl>`) so the outbound send can reconstruct the
  reply target without a side channel.

### Capability flags

| Flag | Value | Notes |
|------|-------|-------|
| `DisplayName` | `Agent 365` | |
| `SupportsStreaming` | `false` | Activity streaming is wired in a later PBI; the loop buffers the full reply. |
| `SupportsInboundImages` | `true` | Image attachments surface as reference content parts for vision models. |
| `SupportsSteering` / `SupportsFollowUp` / `SupportsThinkingDisplay` / `SupportsToolDisplay` | `false` | Not part of the Register tier. |

## Configuration

The adapter binds directly from the `channels:agent365` section (it does **not** use the generic
`Channels.Instances` shape). Option names come from `Agent365GatewayOptions`:

```json
{
  "channels": {
    "agent365": {
      "clientId": "${AGENT365_CLIENT_ID}",
      "clientSecret": "${AGENT365_CLIENT_SECRET}",
      "tenantId": "${AGENT365_TENANT_ID}",
      "channelServiceEndpoint": "https://smba.trafficmanager.net/amer/",
      "agentId": "farnsworth",
      "inboundRoute": "/agent365/messages"
    }
  }
}
```

| Key | Required | Sensitive | Description |
|-----|----------|-----------|-------------|
| `clientId` | yes | no | Entra application (client) ID of the registered Agent 365 app. |
| `clientSecret` | yes | yes | Entra app client secret used to authenticate outbound Activity replies. |
| `tenantId` | no | no | Entra tenant ID. Omit for multi-tenant apps. |
| `channelServiceEndpoint` | no | no | Base URL outbound activities post to. Defaults to the inbound activity's `serviceUrl`. |
| `agentId` | yes | no | BotNexus agent ID inbound messages route to. |
| `inboundRoute` | no | no | HTTP route the message endpoint is hosted on. Defaults to `/agent365/messages`. |

## Microsoft.Agents.* packages and the Microsoft.Extensions.* pin

> **Design note (for maintainers).** The Microsoft 365 Agents SDK packages
> (`Microsoft.Agents.Builder` / `Connector` / `Core` / `Authentication.Msal`, pinned to `1.6.150`)
> depend transitively on the **stable** `Microsoft.Extensions.* 10.0.7` line, whereas BotNexus
> centrally pins the `10.0.0-preview.3` line for the `Microsoft.Extensions.*` packages it references
> **directly**. Because this extension does **not** add a direct `PackageReference` to any
> `Microsoft.Extensions.*` package, Central Package Management imposes no constraint on the SDK's
> transitive `10.0.7` closure and there is **no NU1605 downgrade** (`10.0.7 > preview.3`, so nothing
> is downgraded). Nothing project-references this extension — it is loaded dynamically through the
> `AssemblyLoadContext` extension loader into its own ALC at runtime — so the newer
> `Microsoft.Extensions.*` stays isolated to this leaf and the Cli/Gateway/Scenarios build graphs keep
> the pinned preview line untouched. If a future SDK bump reintroduces a **direct** `Microsoft.Extensions.*`
> reference on this project, expect NU1605 and resolve it with a scoped `VersionOverride` rather than a
> repo-wide pin change.

## Registration

```csharp
services.AddBotNexusAgent365Channel(options =>
{
    options.ClientId = "...";
    options.ClientSecret = "...";
    options.AgentId = "farnsworth";
});
```

The extension is disabled by default in its manifest; enable it by configuring the `channels:agent365`
section and setting the manifest `enabled` flag (or dropping the built extension into the extensions
folder).
