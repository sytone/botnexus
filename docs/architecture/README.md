# BotNexus Architecture (arc42-lite)

**Status:** First slice — architecture-docs foundation (partially addresses #220)
**Scope:** A pragmatic, arc42-inspired overview of the BotNexus platform, grounded in
the real `src/` layout. This document is intentionally concise. Deeper, per-component
arc42 sections, C4 diagram images, and additional ADRs are tracked as follow-ups
(see [ADR index](adr/README.md)).

This overview complements the existing, more detailed references under
[`docs/architecture/`](overview.md) — it is a single entry point framed against the
[arc42](https://arc42.org/) template so that new contributors can orient quickly.

---

## 1. Introduction and Goals

BotNexus is an experimental, domain-driven **multi-agent execution platform** built in
C#/.NET (`net10.0`). A **Gateway** runs within a **World** (a runtime context that
defines which agents, resources, and capabilities are available), orchestrates LLM
interactions, executes tools, and manages persistent sessions across multiple channels.

### Quality goals

| Goal | Why it matters |
|------|----------------|
| **Portability** | Runs on Windows and Linux; no OS-specific assumptions in code or docs. |
| **Testability** | Domain is framework-agnostic; filesystem access via `System.IO.Abstractions`. |
| **Extensibility** | Tools, channels, hooks, and prompts are pluggable extensions. |
| **Streaming-first** | All LLM interactions stream events to clients. |
| **Isolation** | Each (agent, channel) pair gets its own persistent conversation/session. |

### Stakeholders

- **Contributors** building agents, tools, channels, and providers.
- **Operators** running the Gateway and connecting channels (SignalR/WebUI, Telegram, TUI, cron).
- **Agents themselves**, which read repository conventions (`AGENTS.md`) and skills.

---

## 2. Constraints

- **.NET 10 / C#** with central package management (`Directory.Packages.props`) and
  shared build properties (`Directory.Build.props`). See the repo root
  [`AGENTS.md`](../../AGENTS.md) for the authoritative conventions.
- **Cross-platform**: all paths via `Path.Combine()` / `IFileSystem`; CI runs on
  Windows and Linux.
- **Warnings-as-errors** is the target quality bar; tests are mandatory (TDD).
- **Domain purity**: `BotNexus.Domain` has zero external dependencies; the independent
  agent/provider libraries do not depend on BotNexus platform types.

      ┌──────▼───────┐
      │   Channels   │  SignalR (WebUI), Telegram, TUI, cron, Service Bus
      └──────┬───────┘
             │
      ┌──────▼───────┐

      └──────┬───────┘
             │
   ┌─────────▼──────────┐
   │ Agents + Providers │  LLM orchestration (Anthropic, OpenAI, Copilot, ...)
   └─────────┬──────────┘
             │
   ┌─────────▼──────────┐

   └────────────────────┘

- **External systems**: LLM provider APIs (Anthropic, OpenAI, GitHub Copilot,
  GitHub Models, OpenAI-compatible endpoints), messaging channels, and MCP servers.
- **Out of scope for this slice**: detailed sequence diagrams for every flow and full
  C4 diagram imagery.

## 4. Solution Strategy

- **Domain-Driven Design** — `BotNexus.Domain` is the shared vocabulary
  (primitives, value objects generated by Vogen, smart enums, domain models).
- **World as runtime context** — the Gateway operates within a World that defines
  Locations (named resources), agents, and boundaries.
- **Library-backed platform** — agent execution and LLM providers are reusable
  libraries; the Gateway composes them into a platform.
- **Channel-centric routing** — messages route through channels, not direct agent calls.
- **Extension-first** — capabilities are loaded at startup.

## 5. Building-Block View

The following maps to the **actual** top-level folders under `src/` (verified against
the repository layout).

### `src/domain/`
- **BotNexus.Domain** — platform vocabulary. Primitives, value objects, smart enums,
  domain models. Zero dependencies. See [`src/domain/AGENTS.md`](../../src/domain/AGENTS.md).

### `src/agent/`
- **BotNexus.Agent.Core** — agent loop runner, tool execution, hooks.
- **BotNexus.Agent.Providers.Core** — LLM client abstraction, streaming, model registry.
- **BotNexus.Agent.Providers.{Anthropic, OpenAI, OpenAICompat, Copilot, GitHubModels, IntegrationMock}**
  — provider-specific implementations depending only on `Providers.Core`.

### `src/gateway/`
The platform composition root and host. Notable projects include:
- **BotNexus.Gateway** — platform core (supervisor, config, isolation).
- **BotNexus.Gateway.Api** — REST API + SignalR hub host.
- **BotNexus.Gateway.Abstractions / .Contracts** — platform interfaces.
- **BotNexus.Gateway.Channels** — channel plumbing.
- **BotNexus.Gateway.Conversations / .Sessions / .Dispatching / .Prompts** —
  conversation, session, routing, and prompt-pipeline concerns.
- **BotNexus.Gateway.Webhooks** — inbound webhook handling.
- **BotNexus.Gateway.Telemetry / .Telemetry.Abstractions** — observability.
- **BotNexus.Cli** — CLI commands, config management, doctor.
- **BotNexus.Cron** — scheduled triggers.
- **BotNexus.Memory** — agent memory.
- **BotNexus.Tools** — file tool library (read, write, edit, grep, glob, ls).
- **BotNexus.Yaml** — YAML support.

### `src/extensions/`
Dynamically loaded capabilities, including channels and tools:
- **Channels**: `BotNexus.Extensions.Channels.{SignalR, SignalR.BlazorClient(.Core/.Mobile), Telegram, Tui, ServiceBus, Agent365}`.
- **Tools/capabilities**: `BotNexus.Extensions.{ExecTool, ProcessTool, WebTools, Mcp, McpInvoke, Skills, Qmd, DataStore, DebugTool, AudioTranscription}`.
See [`src/extensions/AGENTS.md`](../../src/extensions/AGENTS.md).

### `src/persistence/`
- **BotNexus.Persistence.Sqlite** — SQLite-backed persistence.

## 6. Runtime View (message / streaming flow)

At a high level, an inbound message flows:

1. A **channel** (e.g. SignalR/WebUI, Telegram, TUI, cron) receives input and hands it
   to the Gateway.
2. The Gateway **routes** the message to the target agent's **session** (one per
   agent+channel pair), rehydrating conversation history.
3. The **prompt pipeline** assembles the system prompt and context.
4. **Agent.Core** runs the agent loop against a **provider**, streaming events.
5. **Tool calls** are dispatched to extensions (exec, process, web, MCP, skills, ...)
   and results fed back into the loop.
6. Streamed events flow back out through the channel to the client.

For deeper detail see the existing
[message flow](../development/message-flow.md) and
[LLM request lifecycle](../development/llm-request-lifecycle.md) development docs.

## 7. Deployment View

- **Gateway host** — `BotNexus.Gateway.Api` (ASP.NET) hosts the REST API and the
  **SignalR** hub.
- **WebUI** — a **Blazor Server** presentation surface connects over SignalR for
  real-time chat and agent management.
- **Channels** — Telegram, TUI, Service Bus, and cron connect to the same Gateway.
- **Persistence** — SQLite via `BotNexus.Persistence.Sqlite` for sessions/memory.

The Gateway daemon is managed via the `botnexus gateway` CLI subcommands
(`status`/`start`/`stop`/`restart`).

## 8. Cross-cutting Concepts

- **Value objects** are generated by [Vogen](https://stevedunn.github.io/Vogen/) and
  enforced by architecture fitness tests under
  `tests/architecture/BotNexus.Architecture.Tests/`.
- **Filesystem abstraction** via `System.IO.Abstractions` (`IFileSystem`) for testability.
- **Cross-platform path handling** — always `Path.Combine()`; never hardcoded separators.
- **Telemetry / observability** — see [`observability.md`](../observability.md).
- **Conventions discovery** — agents read nested `AGENTS.md` files and skills.
