## What
Wires config `select` widgets to **dynamic option sources**, so provider fields get real dropdowns instead of free-text. The provider **Default model** field now offers a live model dropdown sourced from `GET /api/models`, and the mechanism generically supports model / thinking / context-size sources.

Fixes Jon's feedback: *"in the providers section there was no drop down for the models, thinking, etc. These should come from the provider's dynamic model list plus anything static."*

## Why
The UI schema (`ConfigSchemaBuilder`) is built by pure static reflection over `PlatformConfig`, so it had no way to inject a provider's live model list — even though `GET /api/models` already returns per-model `SupportedThinkingLevels` / `SupportedContextSizes` (the #1701/#1707 dynamic-capabilities work). `ProviderConfig.DefaultModel` was a plain `Text` widget.

## Changes
- **`ConfigFieldAttribute.OptionsSource`** + `ConfigSchemaBuilder` emits a new `x-ui-options-source` overlay key (stays "annotate once").
- **`ProviderConfig.DefaultModel`** → `Select` with `OptionsSource="models"`.
- **`SchemaForm`**: new `IModelOptionsProvider` abstraction (default `HttpModelOptionsProvider`, backed by `/api/models`, per-provider cached). A select with `x-ui-options-source` resolves options at render time scoped to the enclosing `providers.<key>`; `thinking`/`contextSizes` derive from the sibling-selected model's advertised capabilities; falls back to static `x-ui-options` (enum) when no dynamic list. Leading blank sentinel so optional selects can stay empty.
- **DI**: register `IModelOptionsProvider` in desktop + mobile.
