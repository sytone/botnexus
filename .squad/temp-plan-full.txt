## Executive Summary

Jon's directive fundamentally reshapes our roadmap. The original P2 item — "Design plugin architecture" — is now the foundation that everything else builds on. Channels, providers, and tools must be dynamically loaded from folder-based extension assemblies, referenced by configuration. Nothing loads unless explicitly configured.

This plan re-examines all P0/P1/P2 items through that lens, merges what overlaps, reorders by dependency, and maps every work item to a team member.

**Rev 2 additions:** Jon's directive makes the Copilot provider **P0 — higher priority than all other providers**. Copilot is OpenAI-compatible (base URL: `https://api.githubcopilot.com`) but authenticates via OAuth device code flow, not API key. This introduces new OAuth abstractions in Core and a dedicated `BotNexus.Providers.Copilot` extension project. Provider priority is now: Copilot (P0) > OpenAI (P1) > Anthropic (P2). All work follows conventional commit format.

## Part 1: Dynamic Assembly Loading — Architecture Design

### 1.1 Folder Structure Convention

```
{AppRoot}/
  extensions/
    channels/
      discord/        → BotNexus.Channels.Discord.dll + dependencies
      telegram/       → BotNexus.Channels.Telegram.dll + dependencies
      slack/          → BotNexus.Channels.Slack.dll + dependencies
    providers/
      copilot/        → BotNexus.Providers.Copilot.dll + dependencies
      openai/         → BotNexus.Providers.OpenAI.dll + dependencies
      anthropic/      → BotNexus.Providers.Anthropic.dll + dependencies
    tools/
      github/         → BotNexus.Tools.GitHub.dll + dependencies
```

Each sub-folder is a self-contained deployment unit. It contains the extension assembly and any third-party dependencies not shared with the host (e.g., Discord.Net, Telegram.Bot).

The `extensions/` root path is configurable via `ExtensionsPath` in `BotNexusConfig`. Default: `./extensions` (relative to app base directory).

### 1.2 Configuration Model

The current config has hard-coded typed properties (`ProvidersConfig.OpenAI`, `ChannelsConfig.Telegram`, etc.). These must become **dictionary-based** so the set of extensions is driven entirely by config, not by compiled types.

**New appsettings.json shape:**

```json
{
  "BotNexus": {
    "ExtensionsPath": "./extensions",
    "Agents": { "..." : "unchanged" },
    "Providers": {
      "copilot": { "Auth": "oauth", "DefaultModel": "gpt-4o", "ApiBase": "https://api.githubcopilot.com" },
      "openai": { "ApiKey": "sk-...", "DefaultModel": "gpt-4o", "ApiBase": null, "MaxRetries": 3 },
      "anthropic": { "ApiKey": "sk-ant-...", "DefaultModel": "claude-sonnet-4-20250514" }
    },
    "Channels": {
      "SendProgress": true,
      "SendToolHints": false,
      "SendMaxRetries": 3,
      "Instances": {
        "discord": { "Enabled": true, "BotToken": "...", "AllowFrom": [] },
        "telegram": { "Enabled": true, "BotToken": "...", "AllowFrom": [] }
      }
    },
    "Tools": {
      "RestrictToWorkspace": false,
      "Exec": { "Enable": true, "Timeout": 60 },
      "Web": { "Search": { "Provider": "brave", "ApiKey": "", "MaxResults": 5 } },
      "Extensions": {
        "github": { "Token": "..." }
      },
      "McpServers": { "..." : "unchanged" }
    },
    "Gateway": { "..." : "unchanged" },
    "Api": { "..." : "unchanged" }
  }
}
```

**Key changes:**
- `ProvidersConfig` changes from `{ OpenAI: ProviderConfig, Anthropic: ProviderConfig }` to `Dictionary<string, ProviderConfig>`. Each key is the folder name under `extensions/providers/`. `ProviderConfig` gains an `Auth` discriminator (`"apikey"` default, `"oauth"` for providers like Copilot) so the loader knows whether to expect an API key or delegate to `IOAuthProvider`.
- `ChannelsConfig` keeps its global settings (`SendProgress`, etc.) but moves per-channel config into `Instances: Dictionary<string, ChannelConfig>`. Each key is the folder name under `extensions/channels/`.
- `ToolsConfig` adds `Extensions: Dictionary<string, Dictionary<string, object>>` for dynamically-loaded tool config. Built-in tools (Exec, Web, MCP) remain as-is.
- Root config adds `ExtensionsPath: string`.

### 1.3 Discovery and Loading Process

**Core class:** `ExtensionLoader` (in `BotNexus.Core` or new `BotNexus.Extensions` project)

**Loading sequence (at startup):**

1. **Read config** — Enumerate keys under `Providers`, `Channels.Instances`, `Tools.Extensions`.
2. **Resolve folders** — For each key, compute `{ExtensionsPath}/{type}/{key}/` (e.g., `./extensions/providers/openai/`).
3. **Validate folder** — Confirm folder exists. Log warning and skip if missing.
4. **Load assemblies** — Create an `AssemblyLoadContext` per extension (collectible for future hot-reload). Load all `.dll` files from the folder.
5. **Scan for types** — Search loaded assemblies for concrete (non-abstract) types implementing the target interface:
   - `extensions/providers/*` → `ILlmProvider`
   - `extensions/channels/*` → `IChannel`
   - `extensions/tools/*` → `ITool`
6. **Check for registrar** — If the assembly contains a type implementing `IExtensionRegistrar` (new interface in Core), call its `Register(IServiceCollection, IConfiguration)` method. This lets extensions control their own DI setup.
7. **Fallback to convention** — If no registrar found, register discovered types using `ActivatorUtilities.CreateInstance()`, passing the extension's config section.
8. **Register in DI** — All discovered `IChannel`, `ILlmProvider`, `ITool` instances registered as their interface type (multiple registrations allowed via `IEnumerable<T>`).

### 1.4 New Interfaces

```csharp
// In BotNexus.Core.Abstractions

/// <summary>
/// Optional interface that extension assemblies can implement to control
/// their own DI registration. If absent, the loader uses convention-based
/// registration by scanning for IChannel/ILlmProvider/ITool implementations.
/// </summary>
public interface IExtensionRegistrar
{
    /// <summary>
    /// Registers extension services into the DI container.
    /// </summary>
    /// <param name="services">The service collection.</param>
    /// <param name="configuration">The configuration section for this extension
    /// (e.g., the "openai" section under "Providers").</param>
    void Register(IServiceCollection services, IConfiguration configuration);
}
```

```csharp
// In BotNexus.Core.Abstractions

/// <summary>
/// Interface for providers that authenticate via OAuth instead of API keys.
/// Implementations handle the full OAuth flow (e.g., GitHub device code flow for Copilot).
/// The extension loader skips API key validation for providers that implement this interface.
/// </summary>
public interface IOAuthProvider
{
    /// <summary>
    /// Acquires a valid access token, performing the OAuth flow if needed.
    /// Implementations should cache tokens and handle refresh.
    /// </summary>
    Task<string> GetAccessTokenAsync(CancellationToken cancellationToken = default);

    /// <summary>
    /// Returns true if a cached token exists and is still valid.
    /// </summary>
    bool HasValidToken { get; }
}

/// <summary>
/// Abstraction for secure OAuth token persistence.
/// Implementations may store tokens in OS keychain, encrypted files, or other secure storage.
/// </summary>
public interface IOAuthTokenStore
{
    Task<OAuthToken?> LoadTokenAsync(string providerName, CancellationToken cancellationToken = default);
    Task SaveTokenAsync(string providerName, OAuthToken token, CancellationToken cancellationToken = default);
    Task ClearTokenAsync(string providerName, CancellationToken cancellationToken = default);
}

/// <summary>
/// Represents a cached OAuth token with expiry metadata.
/// </summary>
public record OAuthToken(string AccessToken, DateTimeOffset ExpiresAt, string? RefreshToken = null);
```

### 1.5 Security Considerations

- **Nothing loads by default.** Only assemblies referenced in config are loaded.
- **Folder isolation.** Each extension loads from its own folder — no path traversal.
- **Assembly validation.** Before loading, verify the assembly is a valid .NET assembly. Optionally validate strong-name signatures (configurable, off by default in dev).
- **Interface gate.** Only types implementing known Core interfaces are instantiated. No arbitrary code execution from reflection.
- **Logging.** Every load/scan/register step is logged at Information level. Failures logged at Warning/Error.
- **Isolated AssemblyLoadContext.** Extensions don't pollute the host's type space. Dependency conflicts between extensions are contained.

### 1.6 Development vs. Deployment

**Development (dotnet run / F5):**
- MSBuild targets in each extension project publish output to `{SolutionRoot}/extensions/{type}/{name}/` on build.
- The Gateway project's `ExtensionsPath` defaults to `../../extensions` (relative to output directory) or is overridden via `appsettings.Development.json`.
- Developer adds extension project to solution, builds, config references it, it loads.

**Deployment (dotnet publish):**
- Publish script/target copies extension folders into the publish output under `extensions/`.
- Docker: `COPY extensions/ /app/extensions/`
- Config in production points to `./extensions` (default).

#### `dynamic-assembly-loader`

- **Title:** Build the dynamic extension assembly loader
- **Phase:** 1
- **Priority:** P0
- **Assigned to:** Bender
- **Size:** XL
- **Dependencies:** `config-model-refactor`, `extension-registrar-interface`
- **Description:**
  Implement `ExtensionLoader` in `src/BotNexus.Core/Extensions/` (or a new `BotNexus.Extensions` project if Core should stay pure).

  **Core responsibilities:**
  1. Read `ExtensionsPath` from `BotNexusConfig`.
  2. Enumerate configured extension keys from `Providers` (dictionary keys), `Channels.Instances` (dictionary keys), and `Tools.Extensions` (dictionary keys).
  3. For each configured key, resolve the folder path: `{ExtensionsPath}/{type}/{key}/` where type is `providers`, `channels`, or `tools`.
  4. Validate folder exists. Log warning if missing. Skip gracefully.
  5. Create an `AssemblyLoadContext` (collectible) per extension folder. Load all `.dll` files.
  6. Scan loaded assemblies for:
     - Types implementing `IExtensionRegistrar` → call `Register(services, extensionConfigSection)`

  8. Register all discovered types as their interface types in DI (`IChannel`, `ILlmProvider`, `ITool`).
  9. Log every step: folder scanned, assembly loaded, type discovered, registration completed, errors.

  **Public API:**
  ```csharp
  public static class ExtensionLoaderExtensions
  {
      public static IServiceCollection AddBotNexusExtensions(
          this IServiceCollection services,
          IConfiguration configuration);
  }
  ```

  **Security:**
  - Validate folder paths don't escape the extensions root (no `..` traversal).
  - Only instantiate types that implement known Core interfaces.
  - Catch and log all reflection/loading exceptions — never crash the host.

  **Testing:** Create a test extension assembly (in `tests/`) that implements `IChannel` with minimal behavior. Verify it loads and registers correctly. Verify missing folders are handled gracefully. Verify invalid assemblies don't crash the host.

  **Acceptance criteria:** `AddBotNexusExtensions()` can be called from Gateway `Program.cs`. Extensions in configured folders are loaded and registered. Missing/invalid folders produce warnings, not crashes. Unit tests cover happy path, missing folder, invalid assembly, registrar-based and convention-based loading.

- **Title:** Remove sync-over-async deadlock hazard in MessageBusExtensions

- **Priority:** P0
- **Assigned to:** Farnsworth
- **Size:** S
- **Dependencies:** none
- **Description:**
  The `MessageBusExtensions.Publish()` method in `src/BotNexus.Core/Bus/MessageBusExtensions.cs` wraps an async call with `.GetAwaiter().GetResult()`. This is a deadlock risk in ASP.NET Core's synchronization context.

  **Options (pick one):**
  1. **Delete the method entirely** if all callers can be made async. Search for all call sites of `Publish(` (the sync overload) and replace with `await PublishAsync(`.

  Preferred: Option 1. Grep for all usages and make them async.

  **Acceptance criteria:** No `.GetAwaiter().GetResult()` calls remain in the codebase on async paths. All existing tests pass. Solution builds cleanly.

#### `copilot-provider`

- **Title:** Implement GitHub Copilot LLM provider with OAuth device code flow
- **Phase:** 2
- **Priority:** P0 ⚡ (highest-priority provider — the only one Jon uses)
- **Assigned to:** Farnsworth
- **Size:** L
- **Dependencies:** `provider-dynamic-loading`, `oauth-core-abstractions`
- **Description:**
  Create a new `BotNexus.Providers.Copilot` extension project that connects to GitHub Copilot's LLM API. Copilot uses the OpenAI-compatible HTTP format (same as OpenAI provider) but authenticates via GitHub OAuth device code flow instead of an API key.

  **Reference:** Nanobot's `github_copilot` provider spec:
  ```python
  ProviderSpec(
      name="github_copilot",
      backend="openai_compat",
      default_api_base="https://api.githubcopilot.com",
      is_oauth=True,
  )
  ```

  **Architecture:**
  - The provider implements both `ILlmProvider` and `IOAuthProvider`.
  - HTTP requests use the same OpenAI-compatible format as the OpenAI provider (chat completions, streaming, tool calls). The `LlmProviderBase` retry/backoff logic is reused.
  - The `AsyncHttpClient` sends requests to `https://api.githubcopilot.com` with `Authorization: Bearer {oauth_token}`.
  - Token is obtained via GitHub OAuth device code flow: POST to `https://github.com/login/device/code`, user approves in browser, poll `https://github.com/login/oauth/access_token` until approved.
  - Token is cached via `IOAuthTokenStore`. On each request, check `HasValidToken`; if expired, refresh or re-auth.
  - The Copilot OAuth token grants access to Copilot's model catalog. The default model should be configurable (default: `gpt-4o`).

  **Changes:**
  1. Create `src/BotNexus.Providers.Copilot/` project targeting net10.0.
  2. Add `CopilotProvider : LlmProviderBase, ILlmProvider, IOAuthProvider` — implements ChatAsync/ChatStreamAsync using OpenAI-compatible HTTP format against `https://api.githubcopilot.com`.
  3. Add `GitHubDeviceCodeFlow` — handles the OAuth device code flow: request code → prompt user → poll for token → return access token.
  4. Add `CopilotExtensionRegistrar : IExtensionRegistrar` — registers the provider and its `IOAuthTokenStore` dependency.
  5. Reuse OpenAI provider's HTTP client patterns (request/response DTOs, streaming SSE parsing) — extract shared code to `BotNexus.Providers.Base` if duplication is significant.
  6. Add `CopilotConfig` with `ApiBase` (default: `https://api.githubcopilot.com`), `DefaultModel`, `OAuthClientId`.
  7. Add unit tests with mocked HTTP: chat completion, streaming, tool calling, token refresh, device code flow.
  8. Add MSBuild metadata: `<ExtensionType>providers</ExtensionType>`, `<ExtensionName>copilot</ExtensionName>`.

  **OAuth Device Code Flow detail:**
  ```
  1. POST https://github.com/login/device/code
     → { device_code, user_code, verification_uri, interval }
  2. Display to user: "Go to {verification_uri} and enter code {user_code}"
  3. Poll POST https://github.com/login/oauth/access_token
     with { client_id, device_code, grant_type=urn:ietf:params:oauth:grant-type:device_code }
     every {interval} seconds until token returned or timeout.
  4. Cache token via IOAuthTokenStore.
  ```

## Part 6: Process Guidelines

### Conventional Commits (Mandatory)

All commits must use [Conventional Commits](https://www.conventionalcommits.org/) format. Commit changes as each area of work is completed — not one big commit at the end. This keeps history clean and makes it easy to see what changed and roll back if needed.

**Format:** `<type>(<scope>): <description>`

**Types:**
- `feat` — new feature or capability
- `fix` — bug fix
- `refactor` — code restructuring without behavior change
- `test` — adding or updating tests
- `docs` — documentation only
- `chore` — build, CI, tooling, dependencies
- `perf` — performance improvement

**Scopes** (match project/area): `core`, `config`, `loader`, `gateway`, `bus`, `copilot`, `openai`, `anthropic`, `providers`, `channels`, `tools`, `auth`, `security`, `webui`, `build`, `observability`

**Examples:**
```
feat(core): add IOAuthProvider and IOAuthTokenStore abstractions
feat(config): refactor config models for dictionary-based extensions
feat(loader): implement dynamic assembly loader with ALC isolation
feat(copilot): add GitHub Copilot provider with OAuth device code flow
fix(bus): remove sync-over-async wrapper in MessageBusExtensions
fix(gateway): dispatch to correct agent runner
test(loader): add comprehensive unit tests for extension loader
docs(config): add configuration reference guide
```

**Rule:** Every work item in this plan should result in at least one commit. Large items may produce multiple commits as logical units are completed. The commit scope annotations in each sprint above indicate the expected patterns.

## Decision Needed

This plan requires team consensus before execution begins. Key decisions embedded in this plan:

1. **IExtensionRegistrar approach** — Extensions can optionally provide a registrar class for complex DI setup, with convention-based fallback for simple cases.
2. **WebSocket channel stays hard-coded** — It's core infrastructure, not an extension.
3. **Config model is dictionary-based** — Breaking change from the current typed-property model.
4. **AssemblyLoadContext per extension** — Isolation at the cost of some complexity.
5. **Extensions folder at app root** — `./extensions/{type}/{name}/` convention.
6. **Built-in tools stay in Agent project** — Only external tools are extensions.

**Status:** Awaiting team review and approval.

**New decisions in Rev 2:**

7. **Copilot is P0 provider** — Jon's directive. Copilot provider takes priority over OpenAI (P1) and Anthropic (P2). Sprint 2 leads with Copilot.
8. **OAuth device code flow for Copilot** — Following Nanobot's approach. No API key; runtime acquires token via GitHub OAuth device code flow. Token cached and refreshed automatically.
9. **IOAuthProvider interface in Core** — Providers implementing this skip API key validation. The extension loader and ProviderRegistry must handle both auth modes.
10. **IOAuthTokenStore for secure persistence** — Default implementation uses encrypted file storage. Interface allows future OS keychain integration.
11. **OpenAI-compatible HTTP format reuse** — Copilot uses the same request/response format as OpenAI. Shared HTTP client code should be extracted to Providers.Base to avoid duplication.
12. **Conventional commits enforced** — All agents must use conventional commit format. Commit granularly as work completes, not in bulk.
13. **ProviderConfig Auth discriminator** — Config model needs `Auth` field to distinguish API key vs OAuth providers.
