## What Worked

1. **Parallel agent waves are now routine** — Wave 1 (3 implementation agents) completed in ~17 min, Wave 2 (3 review agents) in ~22 min. Total wall-clock: ~35 min for 7 commits across 6 agents.

2. **WS handler decomposition was clean** — Farnsworth split 409 lines into GatewayWebSocketHandler + WebSocketConnectionManager + WebSocketMessageDispatcher without breaking any existing tests. The façade pattern (keeping HandleAsync) prevented cascading changes.

3. **Nibbler caught a critical provider name mismatch** — CLI was using "github-copilot" as the default provider name while all docs, configs, and code used "copilot". This would have broken every user's first `botnexus init`. Cross-review caught what implementation missed.

4. **Deployment validation proved the dev loop works** — Hermes verified start-gateway.ps1, /health, /webui, /swagger, and auth.json integration all work correctly.

## What Didn't Work

1. **File lock contention between agents** — All 3 Wave 1 agents trying to build/test in parallel caused MSB3021 errors (DLL file locks). Required manual testhost cleanup between waves.

2. **Bender's CLI implementation is monolithic** — 850+ lines of top-level statements in Program.cs. Leela flagged this as P1. Should extract to a Commands/ folder with separate handler classes.

3. **CodingAgent tests are the bottleneck** — 3:42 for 146 tests blocks the entire test suite. Running them separately works but adds friction.

## What to Adjust

1. **Stagger test runs between parallel agents** — Use `--filter` to partition test suites, or have agents skip full-solution tests and let the coordinator run a single validation pass after.

2. **CLI needs structural refactoring** — Before adding more commands, extract the monolithic Program.cs into organized command handlers. This is a P1 carry.

3. **Provider name standardization** — Lock down the canonical provider name convention (use what's in the config, not what's in the code's type names).
